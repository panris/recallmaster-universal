# RecallMaster Universal Research Notes

Date: 2026-05-15

## Tooling Findings

1. RAG evaluation frameworks increasingly separate hard retrieval metrics from LLM-based semantic judgement. Ragas and DeepEval both expose retrieval-centric metrics such as context recall/precision and answer/retrieval faithfulness style checks.
2. Synthetic test generation is useful for scaling coverage, but it should be treated as case authoring, not as final ground truth. The reliable hard metric is still whether a known source ID appears in Top-K.
3. LLM-as-a-judge is strongest for soft dimensions: sub-intent coverage, missing-intent explanation, and noise ratio. It should use consensus mode when possible and route disagreements to manual review.
4. Observability tools such as Phoenix and LangSmith are useful references for trace-first workflows: store every retrieved chunk, judge result, prompt, and run status so a failed case can be replayed.
5. Java integration should prefer a small internal abstraction over direct vendor SDK coupling. PostgreSQL works naturally through JDBC, while Elasticsearch, Chroma, and Milvus can be reached through HTTP APIs or swapped to official clients later.

## Requirement Optimizations

- Replace Python-only Streamlit/Celery assumptions with Java equivalents:
  - Spring Boot WebMVC dashboard and REST API.
  - Java `ExecutorService` plus virtual-thread task orchestration for MVP concurrency.
  - Pluggable judge clients through OpenAI-compatible HTTP endpoints, including LiteLLM/Ollama.
- Make labeling status explicit:
  - `MODEL_PROPOSED`: generated by model/source tracing and not accepted as final.
  - `NEEDS_REVIEW`: model-generated or judge-disputed case requiring human action.
  - `HUMAN_VERIFIED`: accepted hard recall ground truth.
  - `IMPORTED`: trusted external standard set.
- Split metrics by trust level:
  - Hard: `Recall@K` uses `expectedIds`.
  - Soft: intent coverage/noise ratio can be model judged.
  - Governance: unverified labels and judge disagreement always set `needsHumanReview=true`.
- Add RRF as a first-class utility for hybrid retrieval experiments.

## Can Labeling Be Done By Models?

Yes, but only as assisted labeling.

Models can:

- Generate questions and sub-intents from PDF/MD/TXT source documents.
- Suggest target chunk IDs when the generation process is source-traced.
- Judge whether retrieved chunks cover each sub-intent.
- Explain the specific missing intent and propose retrieval tuning actions.

Models should not be the sole authority for:

- Acceptance-grade hard ground truth.
- Regression gates that compare releases.
- Low-frequency, high-impact domain claims such as medical/legal policy details.

The implemented workflow therefore preserves model-produced labels but flags them for review. Production evaluation can still run them, but downstream reports can separate "trusted hard recall" from "synthetic exploratory recall".
